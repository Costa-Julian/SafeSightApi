§Resumen ejecutivo

SafeSight es una plataforma de búsqueda colaborativa de personas desaparecidas que complementa el sistema oficial argentino Alerta Sofía, canalizando la participación ciudadana de forma estructurada y geolocalizada durante las primeras horas posteriores a una desaparición (las llamadas «horas doradas»). La primera entrega definió el problema, la arquitectura propuesta y la justificación tecnológica. Esta segunda entrega documenta el sistema efectivamente construido: el backend, las dos bases NoSQL, el pipeline de datos extremo a extremo y las aplicaciones cliente.

El sistema implementa una arquitectura de persistencia políglota con consistencia eventual: Apache Cassandra absorbe la ingesta masiva de eventos ciudadanos (capa de escritura), mientras que MongoDB sirve como Data Mart pre-agregado para la consulta analítica (capa de lectura). Dos servicios en segundo plano sincronizan ambos motores cada 15 segundos, materializando el mapa de calor que constituye el corazón analítico de la plataforma.

Este documento cubre las rúbricas Modelado de datos inicial (NoSQL) —pendiente de la primera entrega— e Implementación técnica, Pipeline de datos (E2E), Calidad y performance, Ajustes sobre el diseño original y Documentación técnica, correspondientes a la segunda entrega.

1Modelado de datos inicial (NoSQL) Pendiente 1.ª entrega

1.1 Metodología: modelado dirigido por consultas

A diferencia del modelado relacional —que parte de la normalización de entidades y deriva las consultas después—, el modelado NoSQL adoptado sigue el principio query-first: primero se identifican los patrones de acceso y a partir de ellos se diseñan las estructuras de almacenamiento, aun a costa de duplicar datos. Esta inversión es deliberada y se sostiene en dos premisas del dominio:

  • Escrituras dominantes y concurrentes en la ingesta: ante una alerta de alto impacto, decenas de miles de ciudadanos reportan simultáneamente. El modelo de Cassandra se optimiza para absorber ese caudal sin contención.
  • Lecturas analíticas sobre datos heterogéneos: el perfil de la persona, los avistamientos con foto y el mapa de calor requieren estructuras flexibles y pre-computadas, rol que cumple MongoDB.
Tabla 1.1 — Patrones de acceso identificados y motor asignado.
#Patrón de acceso (consulta)FrecuenciaMotor / estructura
Q1Insertar avistamiento/información de un ciudadanoMuy alta (escritura)Cassandra citizen_reports (+ tablas espejo)
Q2Listar todos los reportes de una alertaMediaCassandra citizen_reports
Q3Escanear los reportes nuevos de un día (sync) sin conocer la alertaAlta (batch)Cassandra citizen_reports_by_time

1.2 Modelo en Apache Cassandra — capa de ingesta

Cassandra es la fuente de verdad de los eventos crudos. Su modelo de datos parte del concepto de partition key (determina en qué nodo reside el dato) y clustering key (ordena las filas dentro de la partición). El reto central del dominio es evitar «hot partitions»: una alerta viral concentraría millones de reportes en una sola partición, degradando el rendimiento. La solución es componer la clave de partición con un time_bucket diario (cadena YYYY-MM-DD), que reparte los reportes de una misma alerta entre tantas particiones como días dure su vigencia.

Tabla citizen_reports — tabla maestra (patrón Q1, Q2)

CREATE TABLE citizen_reports (
    alert_id    TEXT,        -- ┐ partition key compuesta:
    time_bucket TEXT,        -- ┘ (alert_id, día) → evita hot partitions
    reported_at TIMESTAMP,   -- ┐ clustering: orden cronológico
    id          UUID,        -- ┘ desempate / unicidad de fila
    type        INT,         -- 0 = Awareness (avistamiento) · 1 = Info
    citizen_id  TEXT,        -- solo Info
    latitude    DOUBLE,
    longitude   DOUBLE,
    description TEXT,         -- solo Info
    photo_url   TEXT,        -- solo Info
    PRIMARY KEY ((alert_id, time_bucket), reported_at, id)
) WITH CLUSTERING ORDER BY (reported_at ASC, id ASC);

Tabla citizen_reports_by_time — espejo para el sync (patrón Q3)

El servicio de sincronización necesita leer «todos los reportes nuevos desde el último checkpoint» sin conocer los alert_id. Como en Cassandra no se puede consultar eficientemente por una columna que no integra la clave, se materializa una segunda tabla cuya partición es únicamente el time_bucket. Es una denormalización deliberada (dual-write): el mismo evento se escribe en ambas tablas en la misma operación.

CREATE TABLE citizen_reports_by_time (
    time_bucket TEXT,                       -- partition key (solo el día)
    reported_at TIMESTAMP, alert_id TEXT, id UUID,
    type INT, latitude DOUBLE, longitude DOUBLE,
    PRIMARY KEY (time_bucket, reported_at, id)
) WITH CLUSTERING ORDER BY (reported_at ASC, id ASC);  -- sin descripción ni foto

Tabla info_reports_sync — staging de reportes con información (patrón Q5)

Solo se escribe cuando type = Info. Conserva los campos completos (descripción, foto, ciudadano) para que el InfoReportSyncService los migre íntegros a MongoDB. Actúa como cola de traspaso entre motores.

CREATE TABLE info_reports_sync (
    time_bucket TEXT, reported_at TIMESTAMP, id UUID,
    alert_id TEXT, citizen_id TEXT, latitude DOUBLE, longitude DOUBLE,
    description TEXT, photo_url TEXT,
    PRIMARY KEY (time_bucket, reported_at, id)
) WITH CLUSTERING ORDER BY (reported_at ASC, id ASC);

Tabla sync_checkpoints — control de avance incremental

Guarda, por cada servicio de sincronización, el último reported_at procesado. Permite lecturas incrementales (procesar solo lo nuevo) y reanudar tras un reinicio sin reprocesar.

CREATE TABLE sync_checkpoints (
    service_name TEXT PRIMARY KEY,    -- 'heatmap_sync' | 'info_report_sync'
    last_processed_at TIMESTAMP
);
Un evento, hasta tres escrituras Un reporte tipo Info se inserta en citizen_reports, citizen_reports_by_time e info_reports_sync; uno tipo Awareness, solo en las dos primeras. La redundancia es el precio de que cada patrón de lectura sea óptimo: en Cassandra se modela por consulta, no por entidad.

1.3 Modelo en MongoDB — datos maestros y Data Mart

MongoDB almacena el dato rico y estructurado y los resultados pre-agregados. El modelo aprovecha el documento embebido para mantener juntas las entidades que siempre se leen juntas (la persona desaparecida y su última ubicación viven dentro del documento de alerta), evitando joins.

Documento heatmap_cells — la unidad analítica

{
  "_id": "{alertId}|{geohash}"  |  "global|{geohash}",   // clave compuesta
  "AlertId": "abc123" | null,                            // null = celda global
  "CenterLatitude": -34.60, "CenterLongitude": -58.38,
  "AwarenessCount": 45, "InfoCount": 8,
  "WeightedIntensity": 69.0,        // = InfoCount*3 + AwarenessCount*1
  "LastUpdated": "ISODate"
}
Diseño del identificador como índice natural El _id de cada celda combina alerta y geohash ("abc123|-34.60:-58.38"). Esto convierte la operación de agregación en un upsert idempotente por clave, sin necesidad de índices adicionales para resolver «¿esta celda ya existe?».

2.2 Arquitectura de componentes

La API es el único punto de contacto con las bases; ningún cliente accede directamente a Mongo o Cassandra. Esto centraliza la lógica de negocio, la validación y la consistencia, y permite que las bases evolucionen (p. ej. hacia un cluster) sin tocar a los clientes.

2.3 Arquitectura en capas y patrones

El backend sigue una separación estricta en tres capas, con inyección de dependencias por interfaces:

   HTTP ─▶ Controller ─▶ Service (interfaz) ─▶ Repository (interfaz) ─▶ BD
           validación        lógica de             acceso a datos
           y envoltura       negocio               (EF Core / CQL)
  • Inyección de dependencias: servicios y repositorios se registran Scoped; los clientes/fábricas de BD y la configuración, Singleton (una única sesión Cassandra compartida, conexión lazy).
  • Contrato uniforme ApiResult<T>: todos los endpoints devuelven { success, data, error }. El manejador global de excepciones emite la misma forma ante un error 500, de modo que el cliente nunca recibe formatos heterogéneos.
  • Paginación PagedResponse<T>: { items, page, pageSize, totalItems, totalPages } para todos los listados.
  • Validación declarativa: un validador FluentValidation por DTO de entrada (rangos de coordenadas, longitudes máximas, fechas no futuras, etc.), aplicado automáticamente antes de llegar al controlador.
  • Serialización: enums como cadena (JsonStringEnumConverter), salvo ReportType que viaja como entero (0/1). Todas las marcas temporales se normalizan a UTC.

2.4 Estrategia dual de acceso a datos

Una decisión central de implementación es usar dos mecanismos de acceso distintos según el motor, en lugar de forzar un ORM único:

MongoDB → EF Core

Mediante SafeSightDbContext con LINQ y OwnsOne para los sub-documentos embebidos. Productividad y tipado fuerte sobre el dato maestro. Se desactivan las transacciones automáticas (AutoTransactionBehavior.Never) porque el Mongo en nodo único (standalone) no las soporta.

Cassandra → driver oficial

Sentencias CQL explícitas vía ISession. Control fino sobre las claves de partición, los dual-writes y las consultas por time_bucket, donde un ORM añadiría una abstracción contraproducente.

2.5 Modelo de roles y reglas de negocio

El dominio distingue dos perfiles en la app móvil, con capacidades deliberadamente asimétricas para mitigar el riesgo legal y las falsas alertas:

Tabla 2.2 — Perfiles, capacidades y mapeo al modelo de datos.

Estado actual — deuda técnica honesta La distinción de perfiles es hoy una regla de la capa cliente y una convención del campo EmitterId: el backend aún no implementa autenticación ni autorización (CORS abierto, sin tokens). Forzar estos roles con un esquema de identidad (p. ej. JWT por perfil) es la principal mejora de seguridad pendiente (ver §4.5 y §5).

3Pipeline de datos extremo a extremo (E2E) 2.ª entrega

El recorrido de un dato desde el dispositivo del ciudadano hasta el mapa de calor del organismo atraviesa cinco etapas. El principio rector es la separación entre el camino de escritura (rápido, sin bloqueo) y el de lectura (pre-computado), unidos por sincronización asíncrona.

 (1) INGESTA                (2) PERSISTENCIA          (3) PROCESAMIENTO        (4) MATERIALIZACIÓN   (5) SERVICIO
 ciudadano                  Cassandra                 BackgroundServices       MongoDB               API → cliente
 ─────────                  ─────────                 ────────────────         ───────               ────────────
 POST /reports/awareness ─▶ citizen_reports        ┐
 POST /reports/info      ─▶ citizen_reports_by_time├▶ lee desde checkpoint  ─▶ heatmap_cells  ┐
                            info_reports_sync (Info)┘   agrupa por geohash      info_reports   ├▶ GET /stats/heatmap
                                                        calcula intensidad                     ┘  GET /reports/info/...
                                                        upsert + checkpoint
                                          cada 15 s  ◀─────────────────────────────────────────

3.1 Etapa de ingesta (escritura)

El ciudadano envía un avistamiento (solo ubicación + timestamp) o un reporte de información (con descripción y foto opcional). La API valida, almacena la foto en disco si corresponde (servida luego como archivo estático bajo /photos/…) y ejecuta el dual/triple-write en Cassandra descrito en §1.2. La respuesta al cliente es inmediata: la escritura no espera a ningún procesamiento analítico.

3.2 Etapas de procesamiento y materialización

Dos BackgroundService de .NET, registrados como hosted services, ejecutan en bucle cada HeatmapSyncIntervalSeconds (= 15 s):

  • HeatmapSyncService: lee de citizen_reports_by_time los reportes con reported_at posterior al último checkpoint; agrupa por geohash (celdas ~1 km²); calcula WeightedIntensity = InfoCount×3 + AwarenessCount×1; hace upsert acumulativo de las celdas por alerta y de las celdas globales en heatmap_cells; y avanza el checkpoint.
  • InfoReportSyncService: migra las filas de info_reports_sync hacia la colección info_reports de MongoDB, con su propio checkpoint independiente.

El geohash implementado para el MVP es un redondeo de coordenadas a dos decimales ("-34.60:-58.38"), que produce celdas de aproximadamente 1,1 km de lado —suficiente para visualización de calor— sin dependencias externas. La distancia geográfica (alcance de una alerta) se calcula con la fórmula de Haversine.

Tabla 3.1 — Trazas del dato a lo largo del pipeline (reporte tipo Info).
PasoDóndeQué ocurreLatencia

El retraso es contractual, no un defecto GET /api/stats/heatmap y GET /api/reports/info/by-alert/{id} leen siempre el resultado pre-agregado en Mongo y están hasta ~15 s por detrás de Cassandra. El heatmap nunca se calcula al vuelo. La lectura de reportes crudos (GET /api/reports/by-alert/{id}) sí va directo a Cassandra, sin retraso.

4Calidad y performance 2.ª entrega

4.1 Modelo de consistencia (CAP en la práctica)

Coherente con la postura AP (Disponibilidad + Tolerancia a particiones) definida en la primera entrega, el sistema materializa la consistencia eventual de forma explícita: el camino de escritura está siempre disponible para no perder eventos en las horas críticas, y la propagación al Data Mart ocurre en segundos. La elección no es una limitación heredada sino una propiedad buscada: el valor de un reporte no depende de que todos los nodos lo conozcan al instante, sino de que no se pierda.

4.2 Pre-agregación como estrategia de performance de lectura

El mapa de calor es la consulta más frecuente y la más costosa de calcular. En lugar de agregarla en cada petición, se materializa incrementalmente en heatmap_cells (patrón Data Mart). El costo de lectura pasa de «escanear y agrupar N reportes» a «leer K celdas ya calculadas», con K ≪ N. El upsert acumulativo por clave {alertId}|{geohash} garantiza que cada ciclo solo procese el delta desde el checkpoint.

4.3 Particionado y prevención de hot partitions

El time_bucket diario en la clave de partición de Cassandra es la principal salvaguarda de rendimiento bajo carga viral: distribuye los reportes de una alerta entre múltiples particiones y nodos, eliminando el cuello de botella de una partición única (§1.2).

4.4 Índices, paginación e idempotencia

  • Índices MongoDB alineados a los patrones de acceso: Status y (EmitterId, Status) en alertas; AlertId en celdas e info-reports; Timestamp descendente en la actividad.
  • Paginación obligatoria en todos los listados, con pageSize acotado (máx. 100) para impedir respuestas no acotadas.
  • Idempotencia en la materialización: los upserts por clave permiten reejecutar un ciclo sin duplicar datos.

4.5 Resiliencia y tolerancia a fallos

La API arranca aunque las bases no estén disponibles: la creación de esquema/índices y la conexión a Cassandra son lazy y registran el error sin abortar el proceso. Los BackgroundService capturan excepciones por ciclo y reintentan en el siguiente, de modo que un fallo transitorio de Mongo no detiene la ingesta en Cassandra. Las notificaciones push degradan con elegancia: si faltan credenciales de Firebase, el envío es un no-op con advertencia en el arranque.

5Ajustes sobre el diseño original 2.ª entrega

La implementación introdujo desviaciones respecto del diseño de la primera entrega. Todas responden a la lógica de un MVP: priorizar lo que demuestra el valor del sistema y diferir la complejidad operativa sin comprometer la arquitectura conceptual, que se mantiene intacta (políglota + consistencia eventual).

6.2 Configuración y despliegue

Requisitos

  • .NET 10 SDK.
  • Docker (para levantar MongoDB en 27017 y Cassandra en 9042).

Puesta en marcha

# 1) Levantar las bases (contenedores de datos, nodo único)
docker compose up -d        # MongoDB + Cassandra en la red safesight-net

# 2) Levantar la API (desde SafeSight.Api/SafeSight.Api/)
dotnet run                  # API en http://0.0.0.0:5121 · Swagger en /swagger

# 3) Clientes
#    · Panel web Next.js  → apunta su baseURL a la API
#    · App Android (Kotlin) → apunta su baseURL a la API (IP de la red local)

Al iniciar, la API crea automáticamente el keyspace y las tablas de Cassandra, los índices de MongoDB y —si las bases están vacías y el seed está habilitado— carga datos de prueba. Migrar las bases entre entornos requiere cambiar únicamente las cadenas de conexión en appsettings.json (o usar appsettings.Docker.json); no hay cambios de código.

§Conclusiones y trabajo futuro

El sistema construido valida la hipótesis central del proyecto: una arquitectura de persistencia políglota con consistencia eventual permite absorber ingesta masiva sin perder eventos y, simultáneamente, servir analítica geoespacial pre-computada con latencia acotada y predecible. La separación entre el camino de escritura (Cassandra) y el de lectura (MongoDB), unida por servicios de sincronización idempotentes con checkpoint, demuestra en la práctica los conceptos de la materia: modelado dirigido por consultas, denormalización deliberada, particionado anti-hot-partition, Data Mart y teorema CAP.

Las desviaciones respecto del diseño original (§5) no comprometen esa arquitectura conceptual, sino que difieren su complejidad operativa hacia iteraciones posteriores. El trabajo futuro prioritario es: (1) introducir identidad y autorización por perfil; (2) escalar a clusters reales de ambos motores; (3) sustituir el geohash propio por indexación geoespacial estándar; y (4) robustecer el acceso a Cassandra con sentencias preparadas y paginación basada en cursor.
