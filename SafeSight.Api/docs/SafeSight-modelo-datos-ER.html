Modelo de datos y relaciones Diagrama entidad–relación lógico sobre persistencia políglota (MongoDB + Cassandra)

1Cómo leer este modelo

SafeSight usa dos motores NoSQL, por lo que no existen claves foráneas físicas que la base imponga (no hay JOINs ni integridad referencial automática). Sin embargo, sí hay relaciones lógicas que la aplicación mantiene por convención. Este diagrama las representa como si fuera un modelo relacional, para hacerlas visibles.

Hay dos ejes de relación:

  • Por alertId (líneas llenas azules): la entidad central es alerts; casi todo lo demás «cuelga» de una alerta. Es una relación 1 : N (una alerta tiene muchos reportes, muchas celdas, etc.).
  • Por id de reporte / linaje (líneas punteadas grises): el mismo reporte ciudadano se escribe con el mismo UUID en varias tablas de Cassandra y luego se migra a Mongo conservando ese id. Es una relación 1 : 1 entre representaciones del mismo evento.

2Diagrama entidad–relación

La clave de todo: alertId Una alerta es el «padre» de todo el resto. Cuando un agente crea una alerta, Mongo le asigna un _id; ese valor se copia como alert_id / AlertId en cada reporte, celda de heatmap, info-report y evento de actividad. No es una FK que la base valide: es la app la que la propaga.

5Linaje: cómo un mismo reporte recorre las tablas

Esto explica la relación 1:1 por id (líneas punteadas del diagrama). Un único reporte ciudadano se materializa con el mismo UUID en varias estructuras:

  El ciudadano envía 1 reporte tipo Info
        │  (la API genera UN id UUID)
        ▼
  CASSANDRA — se escribe el MISMO id en 3 tablas (dual/triple-write):
     · citizen_reports            (lectura por alerta)
     · citizen_reports_by_time    (escaneo por día → heatmap)
     · info_reports_sync          (staging, solo Info)
        │
        │  BackgroundService cada 15s (consistencia eventual)
        ▼
  MONGODB — conservando el MISMO id:
     · info_reports               (Id = id del reporte)        ← InfoReportSyncService
     · heatmap_cells              (agregado por geohash, ya no por id) ← HeatmapSyncService
Dos tipos de "relación" conviviendo Por alertId es una relación clásica padre-hijo (1:N). Por id de reporte es linaje de datos: no relaciona entidades distintas, sino que sigue al mismo dato mientras se copia y transforma entre motores. En heatmap_cells el id del reporte ya se pierde (se agrega por celda), por eso ahí solo queda la relación por AlertId.
