SafeSight · Ingeniería de Datos II · G5

Cómo viaja la información hasta la base Recorrido extremo a extremo de un caso real: «Maitena», desaparición en Merlo, Buenos Aires

Nota sobre los datos Se usa un escenario realista (el nombre Maitena y la localidad de Merlo, Provincia de Buenos Aires) para que los ejemplos no sean «datos fantasma». Los datos personales puntuales (apellido, edad, descripción, coordenadas exactas, identificadores) son ilustrativos y de demostración académica: no se afirman como información verificada de una persona real. Los _id/UUID son ejemplos coherentes entre sí a lo largo de todo el recorrido.

2Un ciudadano confirma «estoy enterado» (avistamiento)

REQUEST · application/json→POST /api/reports/awareness
// el ciudadano solo aporta su ubicación y el momento
{
  "alertId": "6669a1f4c3b2a14d7e8f0021",
  "latitude": -34.6671,
  "longitude": -58.7259,
  "reportedAt": "2026-06-10T16:20:00Z"
}

La API genera un id (UUID) y ejecuta un dual-write en Cassandra. El time_bucket sale de la fecha (2026-06-10). Como es Awareness (type=0), los campos de info quedan en null:

VALUES

   a1b2c3d4-0001-4aaa-9bbb-1c2d3e4f5a60, 0,

4El pipeline materializa los datos en MongoDB (≈15 s después)

Los BackgroundService leen desde su checkpoint y producen el dato listo para consulta. Esto NO ocurre en el request del ciudadano, sino en el ciclo de sincronización.

MONGODB · info_reports (InfoReportSyncService migra desde info_reports_sync, conserva el Id)
// colección: info_reports — lo consume la app para mostrar el detalle
{
  "_id": "b2c1d4e5-0002-4b8c-9d0e-1f2a3b4c5d6e",   // = mismo id del reporte
  "AlertId": "6669a1f4c3b2a14d7e8f0021",
  "CitizenId": "cit-9a4f2b7e",
  "Latitude": -34.6680, "Longitude": -58.7271,
  "Description": "La vi subir a un colectivo de la línea 720...",
  "PhotoUrl": "/photos/c7e0b9a1-3f54-4d8a-9e21-7b6c5d4e3f20.jpg",
  "ReportedAt": "2026-06-10T16:45:00Z"
}
MONGODB · heatmap_cells (HeatmapSyncService agrupa por geohash y acumula)

Ambos reportes (el avistamiento en -34.6671,-58.7259 y la info en -34.6680,-58.7271) caen, al redondear a 2 decimales, en la misma celda -34.67:-58.73 (≈ 1 km²). La celda acumula los conteos de todos los reportes de esa zona para esa alerta:

// colección: heatmap_cells — lo que dibuja el mapa de calor
{
  "_id": "6669a1f4c3b2a14d7e8f0021|-34.67:-58.73",  // {alertId}|{geohash}
  "AlertId": "6669a1f4c3b2a14d7e8f0021",
  "CenterLatitude": -34.67,
  "CenterLongitude": -58.73,
  "AwarenessCount": 37,                 // avistamientos acumulados en la celda
  "InfoCount": 2,                       // reportes con información
  "WeightedIntensity": 43.0,           // = 2×3 + 37×1
  "LastUpdated": "2026-06-10T16:45:12Z"
}

+ se actualiza también la celda global "global|-34.67:-58.73" (mismas coordenadas, AlertId: null), que combina todas las alertas.

El mismo id, distintos roles El reporte de info viaja con el id b2c1d4e5-… por Cassandra y llega a info_reports con ese mismo id (relación 1:1, linaje). En heatmap_cells, en cambio, el id del reporte ya se «disuelve»: la celda solo guarda conteos agregados por zona, no reportes individuales.

5Resumen: qué quedó escrito y dónde

AcciónMotor · estructuraCuándo
Alta de la alertaMongo alerts + admin_activity + disco + FCMen el request
«Estoy enterado»Cassandra citizen_reports + citizen_reports_by_timeen el request
Aporte de informaciónCassandra las 2 anteriores + info_reports_sync + discoen el request
Mapa de calorMongo heatmap_cells (alerta + global)≤ 15 s después (sync)
Detalle de infoMongo info_reports≤ 15 s después (sync)
Hilo conductor Todo se enlaza por alertId = 6669a1f4c3b2a14d7e8f0021. El reporte con información mantiene su id = b2c1d4e5-… desde Cassandra hasta info_reports en Mongo. Las escrituras son inmediatas; la vista analítica (heatmap e info en Mongo) llega con la consistencia eventual de ~15 s.
