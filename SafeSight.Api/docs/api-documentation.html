SafeSight API

Sistema backend para la gestión de alertas de personas desaparecidas. Arquitectura dual NoSQL: MongoDB para alertas (documentos flexibles) y Cassandra para reportes ciudadanos (alta frecuencia de escritura). Sincronización continua mediante un BackgroundService que genera el heatmap geográfico cada 15 segundos.

Stack .NET 10 / ASP.NET Core
Bases de datos MongoDB + Cassandra
Documentación Swagger en /swagger
BASE URL http://localhost:5121
Alerts
Gestión de alertas de personas desaparecidas. Los datos se persisten en MongoDB. Cada alerta contiene información de la persona, ubicación del último avistamiento, estado y el tipo de emisor.

Reports
Reportes ciudadanos almacenados en Cassandra. Diseñados para soportar alta frecuencia de escritura. Existen dos tipos: Awareness (avistamiento simple) e Info (reporte detallado con descripción y foto opcional). Un BackgroundService los agrega en el heatmap de MongoDB cada 15 segundos.
POST /api/reports/awareness Crear reporte de awareness ▼

Registra un avistamiento ciudadano básico: "vi a esta persona aquí en este momento". Es el tipo de reporte más liviano y de alta frecuencia. Solo requiere ubicación y timestamp. Se persiste en Cassandra con partition key compuesta por alert_id y time_bucket.

Body — application/json
{ "alertId": "683b2f4c1a2b3c4d5e6f7890", "latitude": -34.6045, "longitude": -58.3850, "reportedAt": "2026-05-28T14:30:00" }
Campos
CampoTipoDescripción
alertIdrequeridostringID de la alerta asociada al reporte.
latituderequeridodoubleLatitud donde se realizó el avistamiento.
longituderequeridodoubleLongitud donde se realizó el avistamiento.
reportedAtrequeridoDateTimeFecha y hora del avistamiento (ISO 8601).
Respuesta — 200 OK
{ "success": true, "data": { "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890", "alertId": "683b2f4c1a2b3c4d5e6f7890", "type": 0, // 0 = Awareness "latitude": -34.6045, "longitude": -58.3850, "reportedAt": "2026-05-28T14:30:00", "description": null, "photoUrl": null }, "error": null }

Stats
Estadísticas agregadas y datos de heatmap geográfico. El heatmap se genera en MongoDB mediante el HeatmapSyncService que procesa incrementalmente los reportes de Cassandra cada 15 segundos. Cada celda representa una zona de ~1 km².
GET /api/stats/overview Resumen estadístico general ▼

Retorna un snapshot global del estado del sistema: totales de alertas por estado y totales de reportes ciudadanos por tipo.

Respuesta — 200 OK
{ "success": true, "data": { "totalAlerts": 150, "activeAlerts": 42, "resolvedAlerts": 98, "cancelledAlerts": 10, "totalReports": 8742, "totalAwarenessReports": 7900, "totalInfoReports": 842 }, "error": null }
GET /api/stats/heatmap Datos de heatmap geográfico ▼

Retorna las celdas del heatmap geográfico. Sin parámetros devuelve el mapa global (todas las alertas combinadas). Con alertId filtra para una alerta específica, permitiendo visualizar la distribución geográfica de los reportes de una persona. La intensidad ponderada combina awareness (peso 1) e info (peso 3).

Query Parameters
ParámetroTipoDescripción
alertIdopcionalstringSi se provee, filtra el heatmap para esa alerta. Si se omite, retorna el heatmap global.
Respuesta — 200 OK
{ "success": true, "data": { "alertId": null, // null = global, o el ID si se filtró "cells": [ { "centerLatitude": -34.60, "centerLongitude": -58.38, "awarenessCount": 23, "infoCount": 5, "weightedIntensity": 38.0, "lastUpdated": "2026-05-28T14:45:00" } ] }, "error": null }
ℹ La celda del heatmap usa geohash con precisión de 2 decimales (~1 km² por celda). weightedIntensity = awarenessCount × 1 + infoCount × 3.
Notifications
Gestión de tokens de notificaciones push mediante Firebase Cloud Messaging (FCM). En esta versión MVP el token se registra en el log del servidor. En producción se persistirá en base de datos y se usará para enviar notificaciones a dispositivos móviles cuando se creen nuevas alertas.
POST /api/notifications/register-token Registrar token FCM ▼

Registra el token de dispositivo de Firebase Cloud Messaging asociado a un rol de usuario. Utilizado por la app móvil al iniciar sesión para habilitar notificaciones push.

Body — application/json
{ "token": "dGhpcyBpcyBhIGZha2UgRkNNIHRva2Vu...", "role": "citizen" // "citizen" | "entity" }
Campos
CampoTipoDescripción
tokenrequeridostringToken FCM generado por el SDK de Firebase en el dispositivo móvil.
rolerequeridostringRol del usuario: "citizen" o "entity".
Respuesta — 200 OK
{ "success": true, "data": true, "error": null }
ℹ MVP: el token solo se loguea en el servidor. No se persiste ni se usa para envíos reales en esta versión.
