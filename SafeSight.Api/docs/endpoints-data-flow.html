SafeSight API — Endpoints y estructura de datos

Documento orientado al flujo de datos: para cada endpoint se detalla qué recibe, para qué sirve, exactamente qué estructura se escribe o lee en cada base, y qué devuelve. El foco está en la forma de los datos que entran y salen de Cassandra y MongoDB, no en el endpoint en sí.

Arquitectura de datos (leer primero)

SafeSight usa persistencia políglota con dos motores que cumplen roles distintos. Entender cuál base recibe cada escritura es clave para leer este documento.

Cassandra
Capa de ingesta (write-optimized). Fuente de la verdad de los eventos crudos: los reportes ciudadanos. Toda creación de reporte escribe acá. Optimizada para picos masivos de escritura concurrente.
MongoDB
Capa de lectura y análisis (read-optimized). Guarda datos maestros (alerts), datos pre-agregados (heatmap_cells), reportes Info migrados (info_reports), feed de actividad (admin_activity) y tokens (device_tokens).
Consistencia eventual. Dos BackgroundService corren cada 15 s y migran datos de Cassandra a MongoDB:
• HeatmapSyncService: lee reportes nuevos de Cassandra, agrupa por geohash (~1 km²) y hace upsert en heatmap_cells con WeightedIntensity = InfoCount×3 + AwarenessCount×1.
• InfoReportSyncService: migra filas de info_reports_sync (Cassandra) a la colección info_reports (Mongo).
Por eso el heatmap y la lista de reportes Info están ~15 s atrasados respecto al evento original. Es una decisión de diseño, no un defecto.

Envoltorio de respuesta

Todos los endpoints devuelven el mismo envoltorio ApiResult<T>:

{ "success": true,  "data": { ... }, "error": null }
{ "success": false, "data": null,     "error": "Mensaje de error." }

Las listas paginadas devuelven PagedResponse<T> dentro de data:

{
  "items": [ ... ],
  "page": 1,
  "pageSize": 20,
  "totalItems": 45,
  "totalPages": 3
}

GET/api/alerts/{id}/metrics

Métricas de participación ciudadana de una alerta: cantidad de avistamientos, reportes con información y alcance geográfico (distancia máxima desde el último lugar conocido hasta cualquier reporte de tipo Info).

Acceso a base

Mongo · lee
alerts para obtener la alerta y su LastKnownLocation.
Cassandra · lee
Escanea citizen_reports_by_time (últimos 30 días de buckets), filtra por alertId en memoria, cuenta por tipo y calcula la distancia con la fórmula de Haversine.

Responde — AlertMetricsResponse

{
  "alertId": "abc123",
  "totalAwareness": 120,
  "totalInfoReports": 15,
  "geographicReachKm": 4.7
}

Si la alerta no existe: 404.

POST/api/alertsmultipart/form-data

Crea una nueva alerta de persona desaparecida. Si trae foto, la guarda en disco; persiste la alerta en MongoDB; dispara una notificación push y registra un evento en el feed de actividad.

Recibe (form-data) — CreateAlertRequest

CampoTipoReq.Validación
firstNamestringSí≤ 100 car.
lastNamestringSí≤ 100 car.
ageintSí0–120
physicalDescriptionstringSí≤ 500 car.
situationstringSí≤ 1000 car.
latitudedoubleSí-90 a 90
longitudedoubleSí-180 a 180
disappearanceDatedatetimeSíNo futura (ISO 8601)

photofileNojpg/png, ≤ 5 MB

Qué se escribe en las bases

Disco
Si hay photo: se guarda en wwwroot/photos/{guid}.ext y se obtiene la URL relativa /photos/{guid}.ext. PhotoStorageService
Mongo · inserta
Documento en alerts. El _id se genera como ObjectId. Estructura embebida (nota: la persona y la ubicación se anidan, distinto del DTO plano de salida): 
{
  "_id": "ObjectId generado",
  "MissingPerson": {
    "FirstName": "Juan", "LastName": "Pérez", "Age": 35,
    "PhysicalDescription": "...", "PhotoUrl": "/photos/uuid.jpg"
  },
  "Situation": "...",
  "LastKnownLocation": { "Latitude": -34.60, "Longitude": -58.38 },
  "DisappearanceDate": "2026-06-01T10:00:00Z",
  "EmitterId": 1,
  "EmittedAt": "<DateTime.UtcNow al crear>",
  "Status": "Active",
  "ResolvedAt": null
}
Mongo · inserta
Evento en admin_activity con Kind: "AlertCreated", Title y Description generados, Timestamp y AlertId.
FCM
Notificación push fire-and-forget a todos los device_tokens registrados (no bloquea la respuesta).

Responde — 201 Created

AlertResponse con el objeto creado (forma plana, ver GET /api/alerts).

PATCH/api/alerts/{id}/statusapplication/json

Cambia el estado de una alerta. Si el nuevo estado es Resolved o Cancelled, sella ResolvedAt con la hora actual. Registra el cambio en el feed de actividad.

Recibe (JSON) — UpdateAlertStatusRequest

{ "status": "Resolved" }   // "Active" | "Resolved" | "Cancelled"

Qué se escribe en la base

Mongo · actualiza
En alerts: setea Status; si es Resolved/Cancelled también ResolvedAt = DateTime.UtcNow.
Mongo · inserta
Evento en admin_activity con Kind: "StatusChanged".

Responde

{ "success": true, "data": true }. Si la alerta no existe: 404.

Reportes ciudadanos — /api/reports

Toda creación de reporte escribe directo en Cassandra (capa de ingesta). El mismo evento se desnormaliza en varias tablas en una sola operación (ReportsRepository.InsertAsync) para servir distintos patrones de lectura.
POST/api/reports/awarenessapplication/json

El ciudadano indica que está enterado de la alerta enviando solo su ubicación (sin descripción ni foto). Genera un CitizenReport de tipo Awareness.

Recibe (JSON) — AwarenessReportRequest

CampoTipoReq.Validación
alertIdstringSíNo vacío
latitudedoubleSí-90 a 90
longitudedoubleSí-180 a 180
reportedAtdatetimeSíSe convierte a UTC

Qué se escribe en Cassandra

Cassandra · inserta
citizen_reports — fila completa del evento. type = 0, citizen_id / description / photo_url = null. PK ((alert_id, time_bucket), reported_at, id). 
alert_id    = "abc123"
time_bucket = "2026-06-07"        // YYYY-MM-DD de reported_at (UTC)
reported_at = 2026-06-07T14:30:00Z
id          = <UUID generado>
type        = 0
latitude    = -34.603722
longitude   = -58.381592
Cassandra · inserta
citizen_reports_by_time — copia desnormalizada para que el HeatmapSyncService escanee por día sin conocer el alert_id. Solo campos del heatmap: time_bucket, reported_at, alert_id, id, type, latitude, longitude.
Mongo · inserta
Evento en admin_activity con Kind: "ReportSubmitted" ("Nuevo reporte de avistamiento").

Responde — CitizenReport

{
  "id": "uuid", "alertId": "abc123", "type": 0,
  "latitude": -34.603722, "longitude": -58.381592,
  "reportedAt": "2026-06-07T14:30:00Z",
  "citizenId": null, "description": null, "photoUrl": null
}

El ciudadano aporta información concreta con descripción y foto opcional. Genera un CitizenReport de tipo Info. Además de las tablas comunes, escribe una tabla de staging para que se migre a MongoDB.

Recibe (form-data) — InfoReportRequest

CampoTipoReq.Validación
alertIdstringSíNo vacío
citizenIdstringSíNo vacío
latitudedoubleSí-90 a 90
longitudedoubleSí-180 a 180
descriptionstringSí≤ 1000 car.
reportedAtdatetimeSíSe convierte a UTC
photofileNojpg/png, ≤ 5 MB

Qué se escribe en las bases

Disco
Si hay photo: wwwroot/photos/{guid}.ext → photoUrl relativo.
Cassandra · inserta
citizen_reports — fila completa con type = 1 e incluyendo citizen_id, description, photo_url.
Cassandra · inserta
citizen_reports_by_time — copia reducida para el heatmap (sin descripción/foto).
Cassandra · inserta
info_reports_sync — staging exclusivo de tipo Info. El InfoReportSyncService lo lee y lo migra a Mongo. Campos completos:

reported_at = 2026-06-07T14:30:00Z
id          = <UUID>
alert_id    = "abc123"
citizen_id  = "citizen-001"
latitude    = -34.603722
longitude   = -58.381592

photo_url   = "/photos/uuid.jpg"

Evento en admin_activity (Kind: "ReportSubmitted").
Mongo · ~15 s después
El InfoReportSyncService hace upsert del documento en info_reports (esto no ocurre en el request, sino en el ciclo de sync).

Responde — CitizenReport

{
  "id": "uuid", "alertId": "abc123", "citizenId": "citizen-001",
  "type": 1, "latitude": -34.60, "longitude": -58.38,
  "description": "Vi a una persona...", "photoUrl": "/photos/uuid.jpg",
  "reportedAt": "2026-06-07T14:30:00Z"

GET/api/reports/by-alert/{alertId}

Lista todos los reportes (Awareness + Info) de una alerta, del más nuevo al más viejo. Lee directo de Cassandra (datos crudos, sin atraso de sync).

Recibe

ParamUbicaciónDefault

Acceso a base

Estadísticas — /api/stats

GET/api/stats/overview

Resumen general del sistema: conteo de alertas por estado y conteo de reportes por tipo.

Acceso a base

Mongo · cuenta
alerts: total, y conteos por Status (Active/Resolved/Cancelled).
Cassandra · lee
citizen_reports_by_time: trae los reportes (acotado internamente a 30 días de buckets) y los cuenta en memoria, separando Awareness vs Info.

Responde — StatsOverviewResponse

{
  "totalAlerts": 10, "activeAlerts": 6,
  "resolvedAlerts": 3, "cancelledAlerts": 1,
  "totalReports": 540, "totalAwarenessReports": 480, "totalInfoReports": 60
}
GET/api/stats/heatmap

Devuelve las celdas del mapa de calor para renderizar. Siempre lee la colección pre-agregada (Data Mart); nunca calcula al vuelo.

Recibe (query)

ParamTipoNotas
alertIdstringOpcional. Si se omite, devuelve el heatmap global (celdas con AlertId == null).

Acceso a base

Mongo · lee
heatmap_cells filtrando por AlertId (o null para global). Datos mantenidos por el HeatmapSyncService.

Responde — HeatmapResponse

{
  "alertId": "abc123",
  "cells": [{
    "centerLatitude": -34.603722, "centerLongitude": -58.381592,
    "awarenessCount": 45, "infoCount": 8,
    "weightedIntensity": 69.0,
    "lastUpdated": "2026-06-07T14:15:00Z"
  }]
}

weightedIntensity = infoCount×3 + awarenessCount×1.

Referencia — estructuras en Cassandra

Keyspace safesight. La forma exacta de cada fila escrita por la API:

citizen_reports tabla principal de reportes

ColumnaTipoRol / notas
alert_idTEXTPartition key (1/2)
time_bucketTEXTPartition key (2/2) — YYYY-MM-DD (UTC), evita hot partitions
reported_atTIMESTAMPClustering (orden ASC)
idUUIDClustering — unicidad de fila
typeINT0=Awareness, 1=Info
citizen_idTEXTSolo Info (null en Awareness)
latitude / longitudeDOUBLE—
descriptionTEXTSolo Info
photo_urlTEXTSolo Info con foto

citizen_reports_by_time desnormalización para heatmap / scans por día

PK (time_bucket, reported_at, id). Columnas: time_bucket, reported_at, alert_id, id, type, latitude, longitude. No guarda descripción ni foto.

info_reports_sync staging — solo tipo Info

PK (time_bucket, reported_at, id). Campos completos del reporte Info (incluye citizen_id, description, photo_url). El InfoReportSyncService la consume y la lleva a Mongo.

sync_checkpoints control de los background services

service_name TEXT PK + last_processed_at TIMESTAMP. Filas: heatmap_sync, info_report_sync.

Referencia — estructuras en MongoDB

Base de datos safesight.

alerts datos maestros — estructura anidada

{
  "_id": "ObjectId (string)",
  "MissingPerson": { "FirstName", "LastName", "Age",
                     "PhysicalDescription", "PhotoUrl" },
  "Situation": "...",
  "LastKnownLocation": { "Latitude", "Longitude" },
  "DisappearanceDate", "EmitterId", "EmittedAt",
  "Status": "Active|Resolved|Cancelled", "ResolvedAt": null
}

Índices: { Status }, { EmitterId, Status }. Ojo: la API recibe y devuelve la alerta en forma plana (AlertResponse), pero en Mongo se almacena anidada.

heatmap_cells Data Mart pre-agregado

{
  "_id": "{alertId}|{geohash}"  // o "global|{geohash}",
  "AlertId": "abc123" | null,
  "CenterLatitude", "CenterLongitude",
  "AwarenessCount", "InfoCount", "WeightedIntensity",
  "LastUpdated"
}

Lo escribe el HeatmapSyncService (upsert acumulativo). Índice: { AlertId }.

info_reports reportes Info migrados desde Cassandra

{ "_id": "uuid (Guid)", "AlertId", "CitizenId",
  "Latitude", "Longitude", "Description", "PhotoUrl", "ReportedAt" }

Lo escribe el InfoReportSyncService (upsert). Índices: { AlertId }, { CitizenId }.

admin_activity feed del panel

{ "_id": "uuid", "Kind", "Title", "Description", "Timestamp", "AlertId" }

Se escribe en tiempo real al crear alertas/reportes o cambiar estado. Índice: { Timestamp: -1 }.

device_tokens tokens FCM

{ "_id": "ObjectId", "Token", "RegisteredAt" }

Sin índice adicional. El role que llega en el request no se persiste.

Generado a partir del código fuente de SafeSight.Api (controllers, services, repositories y modelos). Las URLs de fotos son relativas (/photos/...) y se sirven como archivos estáticos. Todos los timestamps se almacenan en UTC.
