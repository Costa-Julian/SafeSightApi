Formato general de respuesta

Todos los endpoints devuelven un wrapper ApiResult<T>:

// Éxito
{
  "success": true,
  "data":    /* objeto o array según el endpoint */,
  "error":   null
}

// Error
{
  "success": false,
  "data":    null,
  "error":   "Mensaje de error legible"
}

Las respuestas paginadas usan PagedResponse<T> como valor de data:

{
  "items":      [ ... ],
  "page":       1,
  "pageSize":   20,
  "totalItems": 84,
  "totalPages": 5
}

page     int  default: 1

Respuesta — data
PagedResponse<AlertResponse>

Estructuras en MongoDB

Motor de lectura/consultas. EF Core con driver oficial MongoDB.EntityFrameworkCore 10.0.1. Base de datos: safesight.

alerts Colección MongoDB
Campo Tipo BSON Clave Descripción
_id String PK ObjectId como string generado por la API
MissingPerson.FirstName String Nombre de la persona buscada
MissingPerson.LastName String Apellido
MissingPerson.Age Int32 Edad al momento de la desaparición
MissingPerson.PhysicalDescription String Descripción física (ropa, rasgos, etc.)
MissingPerson.PhotoUrl String Ruta relativa: /photos/{uuid}.jpg
Situation String Texto libre con la situación de la desaparición
LastKnownLocation.Latitude Double Latitud de la última ubicación conocida
LastKnownLocation.Longitude Double Longitud de la última ubicación conocida
DisappearanceDate DateTime Fecha y hora de la desaparición (UTC)
EmitterId Int32 IDX 1 = ciudadano · 2 = entidad (policial, etc.)
EmittedAt DateTime Timestamp de creación de la alerta (UTC)
Status Int32 IDX 0=Active · 1=Resolved · 2=Cancelled
Documento embebido: MissingPerson y LastKnownLocation son subdocumentos embebidos (no colecciones separadas). EF Core los mapea con OwnsOne().

heatmap_cells Colección MongoDB — pre-agregada por HeatmapSyncService
Campo Tipo BSON Clave Descripción
_id String PK Formato: "alertId|-34.60:-58.38" o "global|-34.60:-58.38"
AlertId String? null para celdas globales (todas las alertas)
CenterLatitude Double Latitud del centro de la celda (2 decimales ≈ 1 km²)
CenterLongitude Double Longitud del centro de la celda
AwarenessCount Int32 Cantidad acumulada de reportes tipo "Me enteré"
InfoCount Int32 Cantidad acumulada de reportes tipo "Tengo info"
WeightedIntensity Double InfoCount × 3 + AwarenessCount × 1
LastUpdated DateTime Última vez que el sync actualizó esta celda
Geohash personalizado: se redondean lat/lng a 2 decimales → clave "lat:lng". Una celda cubre aproximadamente 1 km² en latitudes medias. Las celdas se crean y actualizan incrementalmente cada 15 segundos.

Documento de ejemplo — heatmap_cells

{
  "_id":               "6650a1b2c3d4e5f6a7b8c9d0|-34.62:-58.43",

  "CenterLatitude":    -34.62,
  "CenterLongitude":   -58.43,
  "AwarenessCount":    47,
  "InfoCount":         12,
  "WeightedIntensity": 83.0,   12×3 + 47×1
  "LastUpdated":       "2026-05-27T03:00:15Z"
}

Estructuras en Cassandra

Motor de escritura/ingesta. Driver nativo CassandraCSharpDriver 3.22. Keyspace: safesight. Replication: SimpleStrategy (factor 1 para dev; ajustar a NetworkTopologyStrategy en producción).

citizen_reports Tabla Cassandra — particionada por alerta + día
Columna Tipo CQL Clave Descripción
alert_id TEXT PK ObjectId de la alerta en MongoDB
time_bucket TEXT PK Fecha UTC como "YYYY-MM-DD". Evita hot partitions en alertas con muchos reportes
reported_at TIMESTAMP CK Clustering key ASC — permite range queries por tiempo
id UUID CK Clustering key ASC — garantiza unicidad de fila
type INT 0 = Awareness · 1 = Info
latitude DOUBLE Coordenada del ciudadano al reportar
longitude DOUBLE Coordenada del ciudadano al reportar
description TEXT Texto del reporte (solo reportes Info; null en Awareness)
photo_url TEXT Ruta de foto adjunta (opcional, solo Info)
PRIMARY KEY: ((alert_id, time_bucket), reported_at, id) — CLUSTERING ORDER BY reported_at ASC, id ASC
Uso: lectura de reportes de una alerta desde el cliente Android vía GET /api/reports/by-alert/{alertId}.
citizen_reports_by_time Tabla Cassandra — tabla secundaria para el HeatmapSyncService
Columna Tipo CQL Clave Descripción
time_bucket TEXT PK Fecha UTC "YYYY-MM-DD" — permite escanear todos los reportes de un día sin conocer alert_id
reported_at TIMESTAMP CK Clustering key ASC — el sync consulta WHERE time_bucket = ? AND reported_at > ?
id UUID CK Unicidad de fila
alert_id TEXT Referencia a la alerta (para generar celdas por alerta)
type INT 0 = Awareness · 1 = Info
latitude DOUBLE Coordenada para calcular geohash
longitude DOUBLE Coordenada para calcular geohash
PRIMARY KEY: (time_bucket, reported_at, id) — CLUSTERING ORDER BY reported_at ASC, id ASC
Diseño: denormalización deliberada (dual-write). Cada reporte se escribe simultáneamente en citizen_reports y en esta tabla. Solo la usa el HeatmapSyncService; el cliente Android no consume esta tabla directamente.
sync_checkpoints Tabla Cassandra — estado interno del HeatmapSyncService
Columna Tipo CQL Clave Descripción
service_name TEXT PK Identificador del servicio. Valor actual: "HeatmapSyncService"
last_processed_at TIMESTAMP Último reported_at procesado. El sync lee reportes con reported_at > last_processed_at
PRIMARY KEY: (service_name) — tabla simple de un solo registro por servicio. Se actualiza cada vez que el sync termina un ciclo.

Flujo de datos

Escritura de un reporte ciudadano

Android App ──POST /api/reports/awareness──▶ ReportsController ──▶ ReportsService ──▶ Cassandra: citizen_reports (dual-write) ──▶ Cassandra: citizen_reports_by_time (dual-write)

Sincronización al mapa de calor (background, cada 15 s)

HeatmapSyncService ──▶ Cassandra: citizen_reports_by_time WHERE time_bucket = 'YYYY-MM-DD' AND reported_at > last_checkpoint ──▶ agrupa por geohash ("lat:lng") ──▶ upsert MongoDB: heatmap_cells ──▶ UPDATE Cassandra: sync_checkpoints

Lectura del mapa de calor por el cliente

Android App ──GET /api/stats/heatmap?alertId=X──▶ StatsController ──▶ MongoDB: heatmap_cells ◀── lista de HeatmapCellDto
